# Design: Custom Workspace Paths

> Technical approach, touch-points (file:line, verified against the working tree), and the
> one invariant that makes this reversible. Ponytail throughout: reuse the `HERMES_TOOL_ROOTS`
> pathsep-list idiom for the new multi-root config, reuse Compose's native `.env` substitution
> instead of inventing a config format, and change containment checks from "one root" to
> "any of a short list" rather than rewriting them.

## The invariant (hold the line on this)

**Unset everything new → byte-for-byte today's behavior.** No new env var is required. If an
operator never touches the new `.env` keys, `WORKSPACES_DIR` resolves to `${VAULT_DIR}/10_Projects`
exactly as `docker-compose.yml` hardcodes today, `HERMES_BROWSE_ROOTS` resolves to the same single
root `_browse_root()` returns today, and the Add Workspace modal renders exactly as it does now
(no root switcher, no visible change) because there's only one configured root.

## Part 1 — Persisted path config: a repo-root `.env` (ops)

### Grounding (verified)

- `.env` at the repo root is **already gitignored** (`.gitignore:2`) and is Compose's default,
  automatic var-substitution source (`docker compose` loads `./.env` relative to the compose
  file's directory with zero extra flags) — no new tooling, no new format.
- `docker-compose.yml` already reads `VAULT_DIR` via `${VAULT_DIR:-${HOME}/Documents/Obsidian/Vault}`
  at 5 call-sites (`:76,81,85,90,95`) — but nothing ever *sets* `VAULT_DIR` for Compose; it only
  works if the operator exports it in the invoking shell first (undocumented, not persisted).
- `scripts/doctor.sh:6` independently reads `HERMES_VAULT_PATH` (a different name) for its
  host-side secret scan — already inconsistent with the `VAULT_DIR` compose var.

### Approach

- `scripts/install.sh` gains a small, TTY-gated prompt step (skipped entirely under
  `CI`/non-interactive shells — never blocks automation) that writes/updates
  `${REPO_DIR}/.env` (Compose-native, **not** `~/.hermes/.env` — that file stays secrets-only,
  per the existing separation `project-context.md` calls out):
  ```bash
  # ${REPO_DIR}/.env — path overrides (git-ignored). Regenerated by scripts/install.sh;
  # hand-edit is fine, install.sh only fills in missing keys, never overwrites existing ones.
  VAULT_DIR=/Users/you/Documents/Obsidian/Vault
  WORKSPACES_DIR=/Users/you/Documents/Obsidian/Vault/10_Projects
  ```
  Prompts default to today's paths (Enter accepts); an existing `.env` is **never overwritten**,
  only missing keys are appended (idempotent re-run, matches the `.env`-backfill idiom
  `install.sh` already uses for `~/.hermes/.env` at `:70-71`).
- `scripts/install.sh` and `scripts/run.sh` both `mkdir -p` the **resolved** `VAULT_DIR` and
  `WORKSPACES_DIR` before `docker compose up` — the concrete fix for "Docker won't start without
  a real Obsidian vault." A missing bind-mount source is a hard Compose failure today; an
  auto-created empty directory is not (and `vault_graph._scan_notes` already tolerates an empty
  tree, returning `[]` — verified at `engine/gateway/platforms/vault_graph.py:196-217`).
- `scripts/doctor.sh` reads the same `VAULT_DIR`/`WORKSPACES_DIR` (via the repo `.env`, loaded
  with `set -a; . "$ENV_FILE" 2>/dev/null; set +a` before the existing checks) so its host-side
  scan targets the actual configured directory instead of a second, drifting default. One new
  read-only check: both resolved directories exist and are readable (info, not fail, if the
  vault one is a plain empty folder — that's the supported "no Obsidian" state, not an error).
- **Decision — why a `.env` file, not per-workspace or a new JSON config:** the three
  candidates were (a) env vars only (today, not persisted), (b) a dedicated JSON/YAML config
  file, (c) Compose's native `.env`. (b) would need a new loader in both `install.sh` (bash) and
  `docker-compose.yml` (which can't read arbitrary JSON) — extra code for a problem Compose
  already solves. (c) is the smallest rung that holds: zero new parsing, already gitignored,
  already the mechanism these two vars use today, just never written anywhere persistent.

## Part 2 — `docker-compose.yml`: an independent workspaces mount (ops)

### Grounding (verified)

- The `10_Projects` mount today is a **nested sub-mount of the vault** (`docker-compose.yml:91-95`):
  `"${VAULT_DIR:-…}/10_Projects:/vault/10_Projects:rw"`, layered after the vault's own `:ro` mount
  so the more-specific rw mount wins for that subpath (comment at `:79-80`).
- `HERMES_WORKSPACES_ROOT=/vault/10_Projects` is set on the `hermes` service (`:59`) and consumed
  by the `workspace` tool's `artifacts_root` template (`.claude/skills/bmad-agent-workspace/tool.yaml:64`,
  `${HERMES_WORKSPACES_ROOT:-/vault/10_Projects}/{name}/`) via `_root_from_manifest`
  (`api_dashboard.py:548-557`).
- `_browse_root()` (`api_dashboard.py:684-691`) defaults to `_collection_root_for("workspace").parent`
  — i.e. the parent of whatever `HERMES_WORKSPACES_ROOT` resolves to. With the mount unchanged
  from `/vault/10_Projects`, that parent is `/vault` (the whole vault mount) today.

### Approach

```yaml
  hermes:
    environment:
      # unchanged: /vault stays the Obsidian vault mount (Memory/Brain, optional)
      - HERMES_VAULT_PATH=/vault
      - VAULT_DIR=/vault
      # NEW: workspaces get their own mount, independent of the vault
      - HERMES_WORKSPACES_ROOT=/workspaces
      # NEW: pathsep list of roots the folder-picker/ingest may browse+copy from.
      # Mirrors HERMES_TOOL_ROOTS. Default keeps BOTH trees browsable so a project
      # that happens to live inside the vault is still pickable.
      - HERMES_BROWSE_ROOTS=/workspaces:/vault
    volumes:
      - "${HOME}/.hermes:/opt/data"
      - "./.claude/skills:/opt/skills:ro"
      - "${VAULT_DIR:-${HOME}/Documents/Obsidian/Vault}:/vault:ro"
      - "${VAULT_DIR:-${HOME}/Documents/Obsidian/Vault}/30_Resources/Brands:/vault/30_Resources/Brands:rw"
      - "${VAULT_DIR:-${HOME}/Documents/Obsidian/Vault}/30_Resources/Brainstorms:/vault/30_Resources/Brainstorms:rw"
      - "${VAULT_DIR:-${HOME}/Documents/Obsidian/Vault}/30_Resources/Flow:/vault/30_Resources/Flow:rw"
      # REPLACES the old nested "…/10_Projects:/vault/10_Projects:rw" line:
      - "${WORKSPACES_DIR:-${VAULT_DIR:-${HOME}/Documents/Obsidian/Vault}/10_Projects}:/workspaces:rw"
```

Decisions:
- **Default preserves today's exact host path** (`${VAULT_DIR}/10_Projects`) — only the
  **in-container mount point** changes (`/vault/10_Projects` → `/workspaces`). Every workspace's
  `path` field (`workspace.json`, already `additionalProperties: true` in
  `engine/schemas/workspace_state.schema.json`) simply reads `/workspaces/{slug}` instead of
  `/vault/10_Projects/{slug}` after this change — cosmetic in-container detail, not
  host-observable (the host directory is unchanged by default).
- **Brands/Brainstorms/Flow submounts stay vault-anchored** — out of scope (the ask is workspace
  + vault paths, not every artifact root) and they're inherently vault-shaped features
  (Brand Maker, Brainstorm) that assume the PARA vault exists; workspaces is the one tool whose
  output has nothing to do with PARA structure.
- **No `depends_on` change, no healthcheck change** — this is a mount/env change only.

## Part 3 — Engine: multi-root browse + ingest containment

### Grounding (verified)

- `_browse_root() -> Path` (`api_dashboard.py:684-691`) is a single-path return, read by
  `_workspace_browse` (`dashboard/workspace_routes.py:448-483`, containment `target.is_relative_to(root)`
  at `:458`) and by `_ingest_workspace`'s folder-source branch (`api_dashboard.py:763-769`,
  containment `src.resolve().is_relative_to(broot)`).
- `_user_tools_root()`/`HERMES_TOOL_ROOTS` (`api_dashboard.py:560-575`) is the existing precedent
  for a pathsep-separated multi-path env var in this codebase — reused verbatim as the shape for
  `HERMES_BROWSE_ROOTS`.

### Approach

Replace the single-`Path` return with a short list, and change the two containment call-sites
from "is this under **the** root" to "is this under **any configured** root":

```python
def _browse_roots() -> List[Path]:
    """Roots the local-folder picker/ingest may read from, in priority order.
    HERMES_BROWSE_ROOTS (pathsep list, mirrors HERMES_TOOL_ROOTS) takes precedence;
    falls back to the single-path HERMES_BROWSE_ROOT (back-compat); falls back to
    [workspaces root, its parent] so an unconfigured deploy behaves exactly as
    _browse_root() did before this change."""
    multi = os.environ.get("HERMES_BROWSE_ROOTS", "").strip()
    if multi:
        return [Path(p) for p in multi.split(os.pathsep) if p.strip()]
    single = os.environ.get("HERMES_BROWSE_ROOT")
    if single:
        return [Path(single)]
    ws_root = _collection_root_for("workspace")
    return [ws_root, ws_root.parent]
```

- `_workspace_browse`: `root = _browse_roots()[0].resolve()` stays the *default landing* root
  (unchanged response shape: `root/path/parent/entries`), plus a new optional `roots: [str, ...]`
  field (all configured roots) and an optional `?root=` query param to switch the browse base —
  additive, so a client that ignores `roots`/`root` sees identical behavior. Containment
  (`:458`) becomes "matches `target == r or target.is_relative_to(r)` for any `r` in
  `_browse_roots()`", clamping to the *requested* root (or the default) on escape, same as today.
- `_ingest_workspace`'s folder branch (`:763-769`): containment becomes "resolved src is
  relative to any root in `_browse_roots()`" — one loop replacing one `is_relative_to` call, same
  error message shape, now naming the configured roots instead of a single one.
- **No change to `_collection_root_for`/`_root_from_manifest`** — `HERMES_WORKSPACES_ROOT` keeps
  resolving the *destination* independently of the browse/ingest *source* roots; they're
  related-but-separate by design (you can browse from the vault and land in `/workspaces`, or
  vice versa if `HERMES_BROWSE_ROOTS` is configured that way).

Ponytail: one new list-returning function beside the one it replaces (same file, same section),
two call-sites change from a single comparison to a `for root in roots` loop — no new
dependency, no new module, no schema change.

## Part 4 — Dashboard: surface + fallback in "Add Workspace"

### Grounding (verified)

- `NewWorkspaceModal.tsx` folder tab (`:118-132`) renders `<FolderPicker>` with a hardcoded
  caption ("It's copied into 10_Projects.") that is now inaccurate once the mount point is
  `/workspaces`.
- `FolderPicker.tsx` calls `browseFolders(path)` (`:21`) and reads `listing.root`/`.path`/`.parent`/
  `.entries` (`FolderListing` type, `workspaceClient.ts`) — no manual-path input exists; an
  unreachable/errored browse leaves the user stuck (`error` string, no fallback, `:58-59`).

### Approach

- **`workspaceClient.ts`**: extend `FolderListing` with an optional `roots?: { path: string; label: string }[]`
  (present only when more than one root is configured — engine sends it unconditionally, but a
  single-entry array is the common case and callers ignore it) and let `browseFolders(path?, root?)`
  pass the new `?root=` param through.
- **`FolderPicker.tsx`**: when `listing.roots.length > 1`, render a small segmented switcher
  above the path bar (reuses the existing tab-button styling from `NewWorkspaceModal`'s `TABS`
  buttons — no new visual language). When `listing.roots.length <= 1` (today's default), nothing
  new renders — zero visual change for the common case.
- **Manual path fallback**: a `"Type a path instead"` toggle link in the picker footer swaps the
  browser view for a plain text `<input>` (same styling as the existing text inputs in the
  modal), pre-filled with the current `value`; a `"Browse instead"` link swaps back. Submission
  is unchanged (`source_ref` = whatever string is set) — the existing engine-side containment
  check (Part 3) is the only gate, so no new client-side validation is invented. This is the
  concrete answer to "Obsidian not available": if `/vault` isn't mounted from a real vault (or
  the desired folder lives outside every configured root), the user can still name a path
  directly and get the same clear `ingest_failed` error message the API already returns for an
  out-of-bounds path — reused, not re-implemented.
- **Dynamic caption**: replace the hardcoded "It's copied into 10_Projects." with
  `` Copied into the configured workspaces folder (`{short(listing.root)}`). `` sourced from the
  browse response, so it always names the *actual* effective root.

Ponytail: one new optional field on an existing type, one small switcher component reusing
existing button styles, one toggle + one plain `<input>` — no new screen, no new engine
endpoint, no new validation surface.

## Risks & open questions

- **`mkdir -p` on first run creates directories in the user's home without an explicit
  confirmation step beyond the install prompt itself.** Mitigated: `install.sh` already prints
  every resolved path before creating it (the existing `say()` idiom) and the prompt defaults
  are the *exact* paths already implied by the current hardcoded defaults — no surprise new
  location is ever created unless the operator explicitly types one.
  Fine for `WORKSPACES_DIR`/`VAULT_DIR`(scaffold only); it never touches `~/.hermes` (secrets).
- **`HERMES_BROWSE_ROOTS` widens what the "Local folder" tab can read from `/vault` to
  potentially `/workspaces` too** — both are engine-mounted, already-trusted trees (no new host
  surface exposed beyond what Compose already mounts); the security-relevant containment check
  is preserved, just checked against a short admin-configured list instead of one path.
- **Existing workspaces keep working uninterrupted**: their `workspace.json`/marker files store
  `path` as an absolute *in-container* path (`/vault/10_Projects/{slug}` today); after this
  change new workspaces resolve to `/workspaces/{slug}`. Both paths point at the same host
  directory when `WORKSPACES_DIR` is left at its default, so no migration is needed — this is
  called out explicitly in `tasks.md` as a verify step (list/resume/exec of a pre-existing
  workspace after the compose change).
